# Deployment compose

Production-like topology for the Dialysis modular monolith, generated from the .NET
Aspire AppHost (`src/aspire/Dialysis.AppHost`) plus a hand-curated overlay for the few
concerns the Aspire compose publisher doesn't model yet.

## Files

| File | Source | Purpose |
|---|---|---|
| `docker-compose.yaml` | **Generated** by `./build.sh PublishCompose` | Topology — services, env vars, networks, volumes, depends-on edges. Mirrors the AppHost. Do not hand-edit; re-run the NUKE target instead. |
| `.env` | **Generated** by the same target | Container image tags + port placeholders. Operator fills these at deploy time (registry tags + host ports) or leaves them empty when `up --build` provides image names via the override. |
| `aspire-manifest.json` | **Generated** | Aspire's own intermediate description; left next to the YAML for traceability — not consumed at deploy. |
| `docker-compose.override.yaml` | **Hand-maintained** | `build:` stanzas pointing at the repo-root Dockerfiles (so `up --build` works without a registry), host port mappings for the browser-facing services, ASP.NET production hardening (HSTS / forwarded headers / RequireAuthority), Postgres healthchecks, baseline `deploy.replicas`. Treat this as the production complement to the generated topology. |

## Regenerate

After **any** AppHost change (new module, new resource, env-var rename, port change):

```bash
./build.sh PublishCompose --configuration Release
```

The NUKE target wraps `dotnet run --project src/aspire/Dialysis.AppHost --publisher compose
--output-path deploy/compose --deploy false` and tolerates the optional deploy-step failures
(they require a running Docker daemon and aren't relevant for artefact generation). Commit
the regenerated `docker-compose.yaml` + `.env` alongside the AppHost change that produced
them so reviewers see the topology delta inline.

## Run

```bash
cd deploy/compose
docker compose -f docker-compose.yaml -f docker-compose.override.yaml up -d --build
```

This builds every host image from the repo using the existing `Dockerfile.module`,
`Dockerfile.gateway`, and `src/frontend/dialysis-web/Dockerfile`; pulls the infra images
(Postgres, Valkey, RabbitMQ, Keycloak, SonarQube); and brings the whole topology up. The
browser entry point is **`http://localhost:9090`** — same shape as the Aspire dev loop.

Scale a module horizontally:

```bash
docker compose -f docker-compose.yaml -f docker-compose.override.yaml up -d --scale his-api=3
```

Valkey backs the ASP.NET Data Protection key ring so multi-replica is safe.

## Tear down

```bash
docker compose -f docker-compose.yaml -f docker-compose.override.yaml down
# Add -v to wipe the per-module Postgres + SonarQube volumes too.
```

## Why the overlay exists

Aspire's compose publisher emits a deploy-ready topology but doesn't yet model:

1. **`build:` stanzas** — the generated YAML references images via `${X_IMAGE}` env
   vars, expecting a separate build-and-push pipeline. The overlay's `build:` blocks
   point at the existing repo-root Dockerfiles so `docker compose up --build` is
   self-contained.
2. **Host port mappings** — `WithEndpoint(port: 9090, ...)` translates to `expose:`
   rather than `ports:` in the published YAML. The overlay adds the host-side ports
   we always want exposed in deployment (gateway 9090, web 8080, Keycloak 8081,
   SonarQube 9000, per-module Postgres 5440-5445).
3. **ASP.NET production hardening** — `Module__UseHsts`, `Module__UseForwardedHeaders`,
   `Module__Authentication__RequireAuthorityWhenNotDevelopment`, and
   `ASPNETCORE_ENVIRONMENT=Production` are knobs that don't belong on the AppHost
   (which always runs Development) but must be on in deployment.
4. **`ASPNETCORE_URLS` correction** — the AppHost binds the BFF + gateway to
   `localhost` (Aspire's loopback model for dev). In a container they need
   `http://+:<port>` so the compose network can reach them.
5. **Postgres healthchecks** — Aspire's Postgres resource ships a healthcheck for
   dev orchestration but the compose publisher doesn't emit it. The overlay adds
   per-DB `pg_isready` checks so `depends_on: { condition: service_healthy }` would
   work if needed.

Each of these is a candidate to push back into the AppHost (or a custom
`PublishAsDockerComposeService` callback) over time — when something moves, delete
it from this overlay.

## Troubleshooting

| Symptom | Likely cause |
|---|---|
| `docker compose config` fails after a NUKE regen | Aspire model changed in a way the overlay no longer matches (e.g. a service was renamed). Re-read the overlay and patch the changed service name. |
| `docker compose up` hangs on image pull | Docker Hub rate-limited. Authenticate (`docker login`) or pre-pull `nginx:alpine` / `postgres:17-alpine` / `quay.io/keycloak/keycloak:26.0` / `valkey/valkey:9.1`. |
| Module API can't reach Keycloak | Compose network DNS — `keycloak` resolves only inside the compose project. Verify all services are on the `aspire` network. |
| OIDC redirect_uri mismatch | The published YAML uses container hostnames; the realm import (`src/backend/Identity/keycloak/dialysis-realm.json`) registers `http://localhost:9090/*` and `http://localhost:5275/*` — those work via the host port mappings in the overlay. Don't change either side unless the other moves with it. |
