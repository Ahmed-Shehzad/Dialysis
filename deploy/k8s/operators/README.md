# Postgres HA + RabbitMQ HA — Kubernetes operator manifests

These manifests turn the per-module Postgres + the RabbitMQ broker rendered by the
Aspire Helm publisher (`deploy/charts/dialysis-<env>/`) into HA-capable resources.
They're applied **after** the Aspire-generated chart and replace the default
single-pod Postgres / RabbitMQ deployments by routing the same Service names at
operator-managed CRs.

> Why these aren't generated by Aspire: the Aspire 13.4 Kubernetes publisher emits
> per-resource Deployments + Services and doesn't know about CRDs from the Postgres
> or RabbitMQ operators. Rather than carry a brittle post-render step, we ship them
> as hand-maintained YAML that operators apply once per environment.

## What's in here

| File | Purpose |
|---|---|
| `templates/cloudnative-pg-cluster.yaml` | One `Cluster` (postgresql.cnpg.io) CRD per module DB. Primary + N standby with synchronous replication, WAL archiving to S3/MinIO, automatic failover, PITR-ready. |
| `templates/rabbitmq-cluster.yaml` | One `RabbitmqCluster` (rabbitmq.com) CRD. Quorum queues by default (Raft-replicated, disk-flushed across the cluster). |
| `templates/pgbouncer.yaml` | Per-module PgBouncer Deployment + Service in `transaction` pooling mode, fronting the CNPG `Cluster`'s rw Service. |
| `values/<env>.env` | Per-environment knobs (replica counts, sync-replica strategy, storage class, WAL bucket). Source these into `envsubst` to render the templates. |

## Prerequisites

Install the operators once per cluster:

```bash
# CloudNativePG operator (Postgres HA)
kubectl apply --server-side -f https://raw.githubusercontent.com/cloudnative-pg/cloudnative-pg/release-1.24/releases/cnpg-1.24.0.yaml

# RabbitMQ Cluster Operator
kubectl apply -f https://github.com/rabbitmq/cluster-operator/releases/latest/download/cluster-operator.yml
```

These are cluster-scoped controllers; one install per cluster, not per environment.

## Apply order

```bash
ENV=prod                                                       # or dev / staging
NAMESPACE=dialysis-${ENV}

# 1. Make sure the namespace exists (Aspire's chart creates it, but apply order matters)
kubectl create namespace "${NAMESPACE}" --dry-run=client -o yaml | kubectl apply -f -

# 2. Postgres clusters
set -a; source deploy/k8s/operators/values/${ENV}.env; set +a
for f in deploy/k8s/operators/templates/*.yaml; do
  envsubst < "$f" | kubectl apply -n "${NAMESPACE}" -f -
done

# 3. Wait for clusters to be ready
kubectl wait --for=condition=Ready cluster.postgresql.cnpg.io --all -n "${NAMESPACE}" --timeout=10m
kubectl wait --for=condition=AllReplicasReady rabbitmqcluster --all -n "${NAMESPACE}" --timeout=10m

# 4. Install the main Aspire-generated chart — its module Deployments will resolve the
#    Postgres / RMQ Service names to the operator-managed pods.
helm install dialysis deploy/charts/dialysis-${ENV} -n "${NAMESPACE}"
```

## Defense-in-depth story

| Layer | What it guarantees |
|---|---|
| **DurableCommandBus** (`docs/architecture/durable-writes.md`) | Application-layer durability — commands persist in RMQ from the moment of the publisher-confirm ack, applied to Postgres by the consumer in a single explicit transaction (handler change + ledger row commit together) |
| **CNPG sync replication** (here) | Once the consumer commits, the row is mirrored to a sync standby before the transaction returns — Zero RPO at the storage tier |
| **RabbitMQ quorum queues** (here) | Until the consumer applies, the envelope is Raft-replicated across 3 broker nodes. Single-node loss → no impact. Two-node loss → write-unavailable but data preserved. Three-node simultaneous disk loss is the same risk profile as catastrophic loss of the Postgres tier — mitigated by off-cluster backup |
| **WAL archiving** (here, CNPG) | Continuous backup to S3/MinIO. Point-in-time recovery (PITR) bounded by the bucket's retention policy. |

## Connection-string topology

After applying, the in-cluster Service names the Aspire-generated module Deployments
resolve are:

| Module pod env var | Resolves to | Connection path |
|---|---|---|
| `ConnectionStrings__His` | `pgbouncer-his.dialysis-<env>.svc:6432` | → CNPG `pg-his-rw` (primary) |
| `ConnectionStrings__Ehr` | `pgbouncer-ehr.dialysis-<env>.svc:6432` | → CNPG `pg-ehr-rw` |
| `ConnectionStrings__Pdms` | `pgbouncer-pdms.dialysis-<env>.svc:6432` | → CNPG `pg-pdms-rw` |
| `ConnectionStrings__SmartConnect` | `pgbouncer-smartconnect.dialysis-<env>.svc:6432` | → CNPG `pg-smartconnect-rw` |
| `ConnectionStrings__Hie` | `pgbouncer-hie.dialysis-<env>.svc:6432` | → CNPG `pg-hie-rw` |
| `<Module>__Transponder__RabbitMq__ConnectionUri` | `amqp://rabbitmq.dialysis-<env>.svc:5672` | → cluster-managed AMQP endpoint, load-balanced across replicas |

The Aspire-generated chart already emits these env vars pointing at Service names; the
operator CRs match those Service names (`pg-<module>-rw`, `rabbitmq`) so the chart and
the operator CRs compose without per-pod rewiring.

## Failover drills

```bash
# Promote the standby; verify the consumer recovers connection through PgBouncer
kubectl delete pod -n dialysis-prod -l cnpg.io/cluster=pg-pdms,role=primary

# Drop one broker node; verify quorum queues stay writable
kubectl delete pod -n dialysis-prod rabbitmq-server-0

# Both should be no-op at the application layer — durable enqueues continue to succeed,
# consumers reattach automatically, no command_ledger rows are duplicated.
```
