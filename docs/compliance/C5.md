# BSI C5 (Cloud Computing Compliance Criteria Catalogue) — Coverage Map

This document maps the Dialysis modular monolith's code-level controls to the BSI C5 (2020) domains.
C5 is a German cloud-security framework (BSI) widely adopted as a baseline for cloud services serving
German public authorities and the healthcare sector (SGB V §393). The catalogue has 17 domains and
~125 criteria across them.

A C5 attestation is issued by an independent auditor (CPA or specialised IT auditor) as a detailed
report covering either a key date (Type 1) or a 6–12 month period (Type 2). This document is **not**
an attestation; it is the engineering-side map showing which criteria are addressed by code in this
repository versus which are operational (delivered by the cloud platform, deployment process, or
runbook).

## Legend

- **Code** — directly implemented in this repository.
- **Config** — implemented by configuration the operator supplies at deploy time.
- **Operational** — addressed outside the code (cloud-provider attestation, runbook, contracts, audits).

## Horizontal scaling readiness (relevant to BCM + RB)

- **Stateless module hosts.** No in-memory session state. JWT bearer is the only request-time identity carrier.
- **Distributed cache.** `IDistributedCache` backed by Valkey replaces `IMemoryCache` for cross-replica state. Configure per module under `<Module>:DistributedCache:Valkey:ConnectionString` and `AddModuleHost` auto-wires it.
- **Data Protection key ring.** Persisted to Valkey when distributed cache is configured, so any replica can decrypt anti-forgery tokens / encrypted cookies issued by any other replica.
- **Transactional outbox.** Cross-module integration events are written to the module's own Postgres in the same transaction as the aggregate change, then relayed to RabbitMQ by background workers — surviving replica crashes and broker outages.
- **Idempotent consumers.** Transponder consumers track processed message IDs in the inbox table, so re-delivery on replica restart is safe.
- **Per-replica Postgres connection pools.** Each replica pools to its module's database; PG pooling is provider-tunable (`pgbouncer` recommended in front).
- **Liveness + readiness probes.** `/health/live` (process up) and `/health/ready` (DB + cache reachable) — wired in [Dialysis.Module.Hosting.Health](src/backend/Shared/Dialysis.Module.Hosting/Health/) and consumed by the [Dockerfile.module](Dockerfile.module) `HEALTHCHECK`.

To scale a module horizontally:

```bash
docker compose -f docker-compose.yml -f docker-compose.modules.yml up -d --scale his-api=3
```

Each replica binds to the same Valkey + Postgres + RabbitMQ. Place an L7 load balancer (Envoy / Traefik / ALB) in front; do not use sticky sessions.

## What an auditor will still need (operational deliverables, not code)

1. **Cloud-provider attestation report** — for the underlying IaaS (AWS / Azure / GCP all hold C5).
2. **Data location proof** — explicit region pinning in deployment manifests; data-processing agreements (DPAs) with sub-processors.
3. **Personnel records** — background checks, training records for staff with PHI access.
4. **Vulnerability management evidence** — SCA reports, patch cadence, penetration test reports.
5. **Incident response runbook + tabletop exercises** — documented + dated.
6. **Backup / restore drills** — recent restore-rehearsal logs from the managed-Postgres provider.
7. **Independent audit** — engaging a CPA or qualified IT auditor to issue the C5 attestation report.

This repository aims to make those operational deliverables straightforward by getting the
code-level controls right out of the box.
