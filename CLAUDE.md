## Build, run, test

- **SDK**: .NET 10.0.100 (`global.json`, `rollForward: latestFeature`). Central package management is on (`Directory.Build.props` + `Directory.Packages.props`); add new dependencies as `<PackageVersion>` in `Directory.Packages.props`, not per-project versions.
- **Solution file** is `Dialysis.slnx` (XML-format solution, not `.sln`). Tooling takes it directly: `dotnet build Dialysis.slnx`, `dotnet test Dialysis.slnx`.
- **Local dev = the Aspire AppHost. This is the single dev entrypoint:**
  - `dotnet run --project src/aspire/Dialysis.AppHost`
  - One process brings up per-module Postgres (HIS/EHR/PDMS/SmartConnect/HIE/Lab — PDMS is TimescaleDB, the rest stock Postgres) + RabbitMQ + Valkey + Keycloak **and** runs all six module APIs (HIS, EHR, PDMS, SmartConnect, HIE, Lab) + the seven per-context BFFs (his/ehr/pdms/smartconnect/hie/admin/portal) + the Identity BFF + the edge Gateway + the seven per-module Vite SPAs. SonarQube also auto-starts (dev only). The Aspire dashboard (logs/metrics/traces) opens automatically; the OTLP endpoint is injected into every project via `OTEL_EXPORTER_OTLP_ENDPOINT` (`ModuleTelemetryExtensions` falls back to it when `<Module>:Telemetry:OtlpEndpoint` is unset).
  - **Browser entry point is the Gateway: `http://localhost:9090`.** The Gateway is YARP path-prefix routing per context: `/{ctx}/api/*`, `/{ctx}/hubs/*`, `/{ctx}/events/*`, `/{ctx}/identity/*`→ that context's BFF; `/{ctx}/*` (catch-all, low priority) → that context's web SPA; `/identity/*`→ Identity BFF; `/auth/*`→ Keycloak. Contexts are `his`, `ehr`, `pdms`, `smartconnect`, `hie`, `admin` (identity/admin console), `portal` (patient portal). The Identity BFF is pinned to `:5275`, each context BFF to `:5301–5307`, and the Gateway to `:9090` because the Keycloak clients only accept those `redirect_uri`s — do not change these ports without updating `src/backend/Identity/keycloak/dialysis-realm.json` and the Gateway's `appsettings.json` clusters.
  - Keycloak is deliberately **not** a persistent container: `--import-realm` only imports when the realm is absent, so a long-lived container would make `dialysis-realm.json` edits invisible. Re-running the AppHost re-imports.
  - There is intentionally **no** base infra-only `docker-compose.yml` and no per-module `dotnet run` workflow — Aspire owns the dev inner loop. `src/aspire/Dialysis.ServiceDefaults` (`AddServiceDefaults()` / `MapDefaultEndpoints()`) supplies service discovery, HTTP resilience, and the `/alive` probe to every host.
- **Identity-only smoke flow**: `src/backend/Identity/docker-compose.yml` (Keycloak + Postgres for the identity realm; realm `dialysis` auto-imported from `src/backend/Identity/keycloak/`). See `src/backend/Identity/RUNBOOK.md` for the BFF + module-behind-JWT smoke test.
- **Tests**: `dotnet test Dialysis.slnx` runs everything. Persistence is **PostgreSQL everywhere** — there is no EF in-memory provider left; integration/repository tests spin up a **PostgreSQL Testcontainer** (so a running **Docker** daemon is required), and the Transponder bus stays in-memory. Module WAF tests use `ModuleWebApplicationFactory` (one container per fixture); HIE/SmartConnect/EHR share one container per assembly via their `*Postgres`/`*WebApplicationFactory` helpers (each test gets a fresh database). Single-project examples:
  - `dotnet test src/backend/HIS/Dialysis.HIS.Tests/Dialysis.HIS.Tests.csproj`
  - `dotnet test tests/Dialysis.ArchitectureTests/Dialysis.ArchitectureTests.csproj`
  - Single test filter: `dotnet test ... --filter "FullyQualifiedName~SomeTestName"`
- **HIS outbox golden-path test** is gated by `HIS_CI_OUTBOX_E2E=1` and needs real SQL Server + RabbitMQ (run via the `his-outbox-e2e` job in `.github/workflows/build-test.yml`, which is `workflow_dispatch`-only). Default `dotnet test` of the HIS test project runs in-memory only — use `--filter "FullyQualifiedName!~HisOutboxRelayGoldenPathTests"` if you have those services but don't want the golden path.
- **EF migrations**: each module's `<Module>.Persistence` project owns its own `DbContext`, design-time factory, and migrations history table (e.g. PDMS uses `pdms.__ef_migrations`), with schema-per-slice naming (`his_security`, `hie_outbound`, …). The Transponder outbox/inbox/saga tables live on that same `DbContext` under the `transponder` schema — never share a `DbContext` across modules and never add a second outbox.
- **Frontend** = **seven independent per-module SPAs** under `src/frontend/` (`his-web`, `ehr-web`, `pdms-web`, `smartconnect-web`, `hie-web`, `identity-web` [admin console, served at `/admin`], `patient-portal-web` [served at `/portal`]). All are React 18 + Vite 6 + TypeScript + TanStack Query + React Router, npm. Each is standalone (no monorepo workspace; instead a small set of cross-cutting files is **duplicated byte-for-byte** across the apps that need them — `src/shared/lazyPage.ts`, `src/shared/ErrorBoundary.tsx` (+ test), the auth shell (`AuthProvider.tsx`, `authApi.ts`), the durable-command client (`useDurableCommand.ts`, `durableCommandsApi.ts`, `DurableCommandProgress.tsx`, `index.ts`), the toast/theme shells (`toastBus.ts`, `ToastHost.tsx`, `ThemeProvider.tsx`), `eslint.config.js`, and `tsconfig.json` are identical in all seven apps (the auth + durable-command files derive their `/{ctx}` prefix from `import.meta.env.BASE_URL`, so one byte-identical copy serves every context); the `src/features/patients/` helpers (`PatientLabel.tsx`, `patientLoader.ts`, `usePatientName.ts`) are identical across ehr/pdms/identity (+ hie for `PatientLabel.tsx`), while the hie-web and patient-portal-web copies **intentionally diverge** (different backing API; marked by an "Intentionally diverges" header comment). Per-app scripts: `npm run dev` (Vite; dev ports `5331–5337`), `npm run build` (`tsc -b && vite build`), `npm run lint` (`eslint . --max-warnings=0`), `npm run typecheck`, `npm run test:unit` (Vitest), `npm run test:e2e` (Playwright). Husky `pre-commit` + `lint-staged` (eslint + prettier on staged files). Each app proxies `/{ctx}/api`, `/{ctx}/hubs`, `/{ctx}/events`, `/{ctx}/identity`, `/auth` to the Gateway (`VITE_GATEWAY_URL`, default `:9090`) and is reached **via the Gateway origin** under its `/{ctx}` prefix, not its own port directly. The old single `dialysis-web` shell has been removed. Each of the seven apps now ships an `e2e/` folder (Playwright spec + `playwright.config.ts` + fixtures) exercised by the `frontend.yml` CI matrix.
- **SmartConnect PDF SOT**: Python tools under `tools/smartconnect/` (`pip install -r tools/smartconnect/requirements.txt`) regenerate/verify the Mirth user-guide table-of-contents and traceability artifacts. CI (the `pdf-source-of-truth` job in `.github/workflows/docs-regulatory.yml`) requires `docs/book/mirth-connect-user-guide.pdf` materialized via Git LFS — without it the workflow fails by design.
- **Data simulator**: `tools/Dialysis.DataSimulator` is a CLI that drives **id-threaded, cross-module patient journeys** against the running BFFs so the seven SPAs show the *same* related records (one patient flows EHR registration → HIS queue → PDMS session + live vitals → EHR billing → HIE exchange). `tools/Dialysis.DataSimulator/SCENARIOS.md` is the source-of-truth catalog of named verification scenarios (S1 registration, S2 appointment↔encounter, S3 session + vitals, …) that the backend smoke tests and Playwright e2e suites assert against. `Program.cs` runs two loops: `ContinuousDataWorker` (journeys + one-time registry seeds) and `PdmsVitalsTickerService` (streams chairside vitals).

### Versioning & releases

Package versions are **derived from git history by GitVersion — never hand-edited.** The config is `GitVersion.yml` at the repo root (GitVersion v6 schema, GitHub-Flow workflow); the tool is pinned twice and must stay in lockstep: `gitversion.tool` `6.7.0` in `.config/dotnet-tools.json` (for `dotnet gitversion` directly) and a `<PackageDownload Include="GitVersion.Tool" Version="[6.7.0]"/>` in `build/_build.csproj` (for the NUKE build). No project hardcodes `<Version>`.

- **See the current version**: `./build.sh ShowVersion` (or `dotnet tool restore && dotnet gitversion`). This is also the fail-fast gate run first in `build-test.yml` (`./build.sh ShowVersion Test Pack`) — it asserts a version could be derived, so a broken git state fails up front instead of shipping a mis-versioned package.
- **What you get**: `main` between tags → `1.4.1-ci.3`; a `feature/*` or `claude/*` branch → `1.4.1-<branch>.5`; a pull request → `1.4.1-PullRequest0179.5`; a commit **tagged** `vX.Y.Z` on `main` → the clean `1.4.0`. The trunk increments the patch per merge; a `+semver: minor|major` commit-message footer bumps higher.
- **Cut a release**: tag a `main` commit `git tag vX.Y.Z && git push origin vX.Y.Z` — from then on the tag drives the version (the `next-version: 1.0.0` seed in `GitVersion.yml` only applies until the first tag exists). `NUKE Pack` stamps the derived version onto the `.nupkg`s via `/p:Version`.
- **Override**: `./build.sh Pack --version 1.4.0` wins over GitVersion (for an offline build or a manual one-off). CI needs a non-shallow checkout (`fetch-depth: 0`, already set) because GitVersion reads the full history + tags.

## Deployment (containerized stack)

The deployment topology is **generated from the Aspire AppHost** — the AppHost is the single source of truth. There is **no** hand-curated overlay: every concern the Aspire publishers don't model out-of-the-box (build stanzas, host ports, ASP.NET production hardening, the OTLP collector, replica counts, healthcheck cadence) lives in `src/aspire/Dialysis.AppHost/ComposePublishExtensions.cs` and is shaped per environment by `DIALYSIS_DEPLOY_ENV`. A CI drift gate (`deploy-artifacts.yml`) mechanically enforces that the committed `deploy/compose/<env>/` and `deploy/charts/dialysis-<env>/` match what the AppHost regenerates.

**Two publishers, three environments.**

```bash
# Docker Compose — one folder per environment
./build.sh PublishCompose      --environment prod   # → deploy/compose/prod/
./build.sh PublishAllCompose                        # fan out across dev / staging / prod

# Kubernetes / Helm — one chart per environment
./build.sh PublishKubernetes   --environment prod   # → deploy/charts/dialysis-prod/
./build.sh PublishAllKubernetes                     # fan out across dev / staging / prod
```

Each NUKE target wraps `dotnet run --project src/aspire/Dialysis.AppHost --publisher {compose|k8s} --output-path … --deploy false`. Always pass `--deploy false` — without it Aspire tries to talk to a Docker daemon during publish. After **any** AppHost change, re-run the appropriate `PublishAll*` and commit the regenerated artifacts alongside the AppHost edit so the three env shapes stay in lockstep (the `deploy-artifacts.yml` job fails the PR otherwise).

The per-env shapes share ~90% of the topology; the diff is a small set of strings (`ASPNETCORE_ENVIRONMENT`, HSTS / `ForwardedHeaders`, `Authentication__RequireAuthorityWhenNotDevelopment`, OTLP wiring, replica counts, healthcheck cadence, host port mappings). See `deploy/compose/README.md` for the per-env matrix.

**Image registry (JFrog Artifactory / any OCI registry)**: `./build.sh PushImages --registry <host>/<docker-repo>` builds all 22 repo-built images (module APIs, BFFs, gateway, SPAs), pushes them tagged with the GitVersion SemVer, and writes `artifacts/images/values-images-<env>.yaml` for `helm install -f`. The committed `deploy/` artifacts stay registry-free (the drift gate regenerates them with a bare environment); registry qualification is publish-time-only via `DIALYSIS_IMAGE_REGISTRY`/`DIALYSIS_IMAGE_TAG`. See `docs/operations/container-registry.md`.

Run a compose env locally:

```bash
cd deploy/compose/prod
docker compose up -d --build
```

This builds every host image from the repo using `Dockerfile.module` (parameterised by `MODULE_PROJECT`/`MODULE_DLL`), `Dockerfile.gateway`, and per-app `Dockerfile`s under each `src/frontend/<app>/`. Module APIs (compose host ports): HIS `:5288`, EHR `:5289`, PDMS `:5290`, SmartConnect `:5291`, HIE `:5292`, Lab `:5293`; Identity BFF `:5275`; per-context BFFs `:5301–5307`; Gateway `:9090`; the seven web apps run behind the Gateway. Module hosts are stateless and scale horizontally (`docker compose up -d --scale his-api=3`); Valkey backs the distributed cache, the BFF session ticket store, **and** the ASP.NET Data Protection key ring, so multi-replica is safe. CI (`.github/workflows/*`) uses plain `dotnet`; it does **not** build these images.

For Kubernetes, the Helm chart at `deploy/charts/dialysis-<env>/` is renderable on any cluster — `helm install dialysis ./deploy/charts/dialysis-prod`. The gateway endpoint is marked external so the chart emits an `Ingress` for it.

## Architecture

A **modular monolith**: each bounded context under `src/backend/<Module>/` (HIS, EHR, PDMS, SmartConnect, HIE, Lab, Identity, plus the headless `PatientPortal` BFF) has its own ASP.NET host(s) and its own database. Modules communicate **only** via integration events over Transponder (RabbitMQ in the deployment stack, in-memory in dev/tests) — never via direct domain references.

### Event storming, not event sourcing

We model with **event storming** (Brandolini): commands, aggregates, policies, domain events, integration events, read models. We **do not** implement event sourcing — aggregates persist current state via EF, never as a replayable log. The split in code:

- **Domain events** (`Dialysis.DomainDrivenDesign.DomainEvents.IDomainEventHandler<T>`) — in-process, raised by aggregates, dispatched within the same `SaveChanges` transaction. Use for *within-context* coordination.
- **Integration events** (`Dialysis.DomainDrivenDesign.IntegrationEvents.IIntegrationEvent`) — persisted to the Transponder outbox in the same transaction as the state change, then relayed asynchronously over RabbitMQ. Use for *cross-context* signals. Schema-versioned (`int SchemaVersion`); a bump goes through `IntegrationEventVersioningTests`.
- **Read models** — denormalized tables or in-memory projections built from current aggregate state (e.g. `EfManagerDashboardReadModel`, `EfPatientSearchReadModel`). Never rebuilt from an event log.

Do not introduce `IEventStore`, aggregate-from-event-log, or replay loops. If a feature feels like it wants event sourcing, model it as an aggregate + projection instead.

### Module project layout (HIS is the reference shape)

For module `X`:
- `Dialysis.X.Contracts` — integration events, permission catalog (`XPermissions`), `IPermissionedCommand`, public DTOs. **The only assembly other modules may reference.**
- `Dialysis.X.<Slice>` — vertical slices (e.g. `PatientFlow`, `Scheduling`, `Medication`, `Registration`, `PatientChart`, `TreatmentSessions`) holding commands/queries/handlers/domain types for one bounded slice. Slices also carry a `Fhir/` folder with their FHIR resource mappers.
- `Dialysis.X.Persistence` — single `XDbContext`, schema-per-slice naming, repositories, read models, audit. Extends Transponder's EF persistence base so outbox/inbox/saga live on the same `DbContext`.
- `Dialysis.X.Composition` — single `AddX(...)` registration extension consumed by the host.
- `Dialysis.X.Api` — ASP.NET host. The host bootstrap is an explicit `public partial class Program` with `public static async Task Main(string[] args)` (not top-level statements) so the `WebApplicationFactory`-based test fixtures can target `Program`. Versioned MVC under `api/v1.0/...` via `Asp.Versioning.Mvc` (+ ApiExplorer, URL-segment reader). OpenAPI per group (`GET /openapi/v1.json`). Successful JSON bodies use the HATEOAS envelope `{"data":...,"links":[...]}` (`ResourceEnvelope<T>`). Health: `/health/live`, `/health/ready`.
- `Dialysis.X.Bff` — per-context Backend-for-Frontend (OIDC + path-scoped cookie + bearer-forward YARP proxy to the API + optional event push). See "Per-context BFFs" below.
- `Dialysis.X.Tests` — xUnit + `WebApplicationFactory` integration tests.

**Known, intentional divergences from the reference shape:**
- **Per-module persistence-provider split**: EHR, PDMS, SmartConnect, and HIE have a `Persistence/` sibling folder with `Core.Persistence.{Abstractions,InMemory,Postgresql}` projects (provider chosen at composition time) plus `Core` / `Core.Abstraction` projects. HIS and Lab keep a single `Persistence` project.
- **Lab is a headless module**: `Dialysis.Lab.Api` exists, but Lab has **no** BFF or web app — its reads are surfaced through the EHR/clinical workflow and it talks to the world purely via integration events.
- **PatientPortal is BFF-only**: `src/backend/PatientPortal` contains only `Dialysis.PatientPortal.Bff` — no API, no DbContext, no domain. It's an aggregation/event-push BFF over EHR/PDMS/HIS/HIE for the patient SPA; the patient-facing *domain* (secure messages, appointment requests, after-visit summaries) lives in the EHR `PatientPortal` slice.
- **SmartConnect uses a channel/flow model** (Mirth-style integration engine), not classical DDD aggregates — its sub-areas are `Inbound`, `Adapters`, `Dicom`, `AvScanning`, `Management`.

### Module slice inventory

- **HIS** (`src/backend/HIS/`, reference shape, single `Persistence`, mapped to Tummers et al. 2021): `PatientFlow`, `Scheduling`, `Medication`, `Security`, `RaCapabilities` (reference-architecture capabilities, schema `his_ra`), plus newer slices: `Operations` (facility ops — `StaffMember`, `InventoryItem`, and the `BillingExportJob` queue EHR consumes to file claims; commands `SubmitBillingExportJob`/`ExecuteBillingExportJob`/`RecordInventoryMovement`), `Integration` (RPM device registry + telemetry ingest governance — `Device` lifecycle, `DeviceReadingRecord`, commands `RegisterDevice`/`BindDeviceToPatient`/`IngestDeviceReading`/`ChangeDeviceStatus`), `DataServices` (bulk-import staging `DataImportJob`, patient-search indexing, outbox visibility, manager dashboards), `PatientAccess` (query-only patient-claim-filtered portal summary, `GetPatientPortalSummary`).
- **EHR** (`src/backend/EHR/`, provider-split persistence): `Registration` (Patient root), `PatientChart`, `Scheduling`, `Integration`, plus `Billing` (full claim lifecycle — `Charge`/`Claim`/`Remittance`/`Payment`/`Payer`/`CptFeeScheduleEntry`, EDI 837/277CA/999, charge-edit blocking via `IChargeEditChecker`; commands `CaptureCharge`/`SubmitClaim`/`RecordRemittance`/`PostPayment`), `ClinicalNotes` (encounters + documentation + orders — `Encounter`/`ClinicalNote`/`Prescription`/`LabOrder`/`ImagingOrder`/`Diagnosis`/`Referral`/`OrderSet`, `ClinicalSafetyChecker`, `QualityMeasureEvaluator`; commands `StartEncounter`/`DraftClinicalNote`/`SignClinicalNote`/`OrderLabTest`/`RequestReferral`/`CloseEncounter`), `PatientPortal` (`SecureMessage` threads, `PortalAppointmentRequest` approve/decline, `AfterVisitSummary`).
- **PDMS** (`src/backend/PDMS/`, TimescaleDB, provider-split persistence): no longer single-slice. `TreatmentSessions` is the core (`DialysisSession`/`TreatmentAlarm`, Evans' *System Metaphor* "a treatment-machine cycle observed through telemetry"), joined by `Medications` (chairside MAR `MedicationAdministrationRecord`, `IvPumpInfusion` + vendor pump drivers Alaris/Baxter/Hospira), `OnCall` (`OnCallRotation`/`EscalationPolicy`/`AlarmDispatch` with per-attempt audit), and `Reporting` (post-session `SessionReport` — discharge letters / shift reports / billing docs via Mustache templates).
- **HIE** (`src/backend/HIE/`, FHIR R4 / IHE — see dedicated section below). New `Query` slice = pull-side partner queries (`PullOutsideRecords`/`PullPartnerRecords`/`PullPartnerDocuments`, XCA document queries with Polly retry).
- **SmartConnect** (`src/backend/SmartConnect/`, channel/flow integration engine): `Inbound` (HL7v2/FHIR ingestion), `Adapters`, `Dicom` (DICOM C-STORE/C-FIND DIMSE + DICOMweb `/dicom-web` + pluggable AI inference gate `IImagingInferenceProvider`), `AvScanning` (ClamAv attachment scanning), `Management` (operator admin API `/api/v1/admin/*` — flow CRUD/lifecycle, ledger reprocessing, alert rules, retention pruning).
- **Lab** (`src/backend/Lab/`, headless): `Orders` slice — `LabOrder` aggregate (Placed→Transmitted→InProgress→Resulted), LOINC-coded; publishes `LabOrderPlacedIntegrationEvent` (SmartConnect transmits to the LIS), consumes `LabResultReceivedIntegrationEvent` (records observations). DbContext `LabDbContext`, schema `lab`. Composition `AddLaboratory()`.

### HIE module (FHIR R4 / IHE health-information exchange)

`src/backend/HIE/` is the cross-organization gateway. It still obeys the module pattern (owns `HieDbContext`, schemas `hie_outbound`/`hie_inbound`/`hie_consent`/`hie_xds_registry`/`hie_documents`/`hie_tefca`/`transponder`; references only other modules' `*.Contracts`). Slices:
- **Outbound** — consumes upstream integration events, maps them to FHIR resources (Patient, Encounter, LabOrder/Result, AdverseEvent, DialysisSession, ClinicalNote), validates against US Core, dispatches to partner endpoints with Polly retry, TEFCA IAS JWT auth, FHIR `AuditEvent` trail, and Pending→Dispatching→Delivered/DeadLettered state.
- **Inbound** — partners `POST /fhir/{Type}`; TEFCA middleware validates the IAS JWT/trust anchor, the ingestion service validates profile + consent, then enqueues `Hl7FhirResourceReceivedIntegrationEvent` for the owning module to consume.
- **Query** — pull-side: FHIR-based patient discovery + record retrieval from partners and XCA document queries (`PartnerFhirQueryClient`, `PartnerPatientDiscoveryClient`, `XcaDocumentClient`), all behind Polly retry.
- **Consent** — `ConsentPolicy` aggregates + `IsResourceAccessPermittedQuery`; EHR/HIS call this as a cross-module query to gate resource visibility.
- **Documents** — `DocumentReference` aggregate + `IDocumentBlobStore`-backed binary content; PAdES signing, server-side AcroForm value fill (`POST /api/v1.0/documents/{id}/fill` via `IAcroFormProcessor.FillFormValuesAsync` — baked into the bytes so a subsequent signature actually covers the filled values), inline preview envelope (`GET /api/v1.0/documents/{id}/preview` — PDFs go through pdfjs against `/binary`; XML / CDA / FHIR-XML are pretty-printed server-side; plain text verbatim; everything else returns the `Binary` hint for download-only). PDF JavaScript stays inert in the viewer by default: `DocumentReference.AllowJavaScriptExecution` is a per-document gate flipped via `POST /api/v1.0/documents/{id}/javascript-execution` (retention-admin permission, audited via `[PhiAccess]`); the SPA passes the flag through to pdfjs `enableScripting`. Owns the GDPR Art. 5(1)(e) retention pipeline: `DocumentRetentionPolicy` per `DocumentReference.Kind`, `RetentionPurgerHostedService` (24-hour tick, opt-in via `Documents:Retention:AutoPurge`), and the Art. 17 `HieDocumentsPatientEraser` that walks every `Current` document for the patient.
- **Tefca** (US TEFCA QHIN onboarding — distinct from the German gematik TI building block at `BuildingBlocks/Tefca/Dialysis.BuildingBlocks.Tefca.Ti`, naming overlap is acronym-only) — `QhinPartner` aggregate (`Onboarding` → `Active` → `Suspended`, activation requires ≥1 trust anchor **and** mTLS material), `TrustAnchorParser` (PEM via `X509Certificate2.CreateFromPem`), `HmacIasJwtIssuer` for IAS JWT minting, admin surface at `/hie/admin/tefca/partners`.
- **Xds** (IHE XDS Registry/Repository, ITI-18/41/42/43) and **OpenEhr** (openEHR Composition ↔ FHIR — projections are now **declarative**: JSON archetype definitions in `Dialysis.HIE.OpenEhr/Archetypes/Definitions/` are evaluated by a `ResourcePath` walker; add a new archetype by dropping a JSON file, no code change required).

### Cross-cutting building blocks (`src/backend/BuildingBlocks/`)

- **Intercessor** — in-process mediator for command/query dispatch.
- **Verifier** — request/command validation pipeline.
- **Transponder** — distributed messaging (`ITransponderBus`); transports (RabbitMQ, NATS, Azure Service Bus, SQS, gRPC, SignalR, SSE); EF outbox/inbox/saga persistence (`Postgresql`, `Shared`); schedulers (Hangfire / Quartz / TickerQ — exactly one per host). The module-hosting scaffolding wires **PostgreSQL-backed Hangfire by default** (`AddTransponderHangfire(...)`) whenever a `Hangfire` connection string is present, so module hosts get background scheduling without per-project setup. `enableOutboxRelay` (or `<Module>:Transponder:EnableOutboxRelay`) opts a host into background publishing. See `src/backend/BuildingBlocks/Transponder/README.md`.
- **Fhir** — ~18 projects implementing the FHIR stack: `Core` (`IFhirResourceMapper<TEvent,TResource>`), `Validation` (US Core), `Smart` (SMART-on-FHIR), `Subscriptions`/`BulkData`/`Audit` (each with an `*.EntityFrameworkCore` persistence project), `Tefca` (IAS JWT + trust anchors), `Terminology`, `DeIdentification`, `OpenEhr`, `CdaBridge`, `AspNetCore` (middleware), `Testing`. Per-module mappers live in each slice's `Fhir/` folder; SmartConnect's `Hl7V2ToFhirPipeline` routes HL7v2 by MSH-9 trigger to per-trigger mappers.
- **DataProtection** — GDPR/BDSG compliance surface. Mounts `/api/v1.0/data-subject-rights/...` via `MapEuDataProtectionRoutes()`. Two per-module participation hooks: `IModuleDataExtractor` (Art. 15 export) and `IPatientEraser` (Art. 17 erasure). The `DefaultDataSubjectRightsService` orchestrates request → approval → execution and writes the audit row through `IErasureRequestStore` (HIE registers a Scoped EF impl; non-HIE hosts fall back to the `InMemoryErasureRequestStore` registered via `TryAddSingleton`). Sibling concerns: consent, lawful bases, RoPA, retention, breach notification, encryption, audit.
- **Hipaa** (`Dialysis.BuildingBlocks.Hipaa` + `.AspNetCore`) — HIPAA Security Rule (45 CFR 164.3xx) scaffolding wired in one call via `AddHipaaCompliance(moduleSlug)` (or granular `AddHipaaPhiEncryption()`/`AddHipaaAuditPipeline()`/`AddHipaaSafeguards()`). Three pillars: column-level PHI encryption (`IPhiProtector`), an audit pipeline that emits FHIR `AuditEvent`s for any CQRS request marked `[PhiAccess(action, fhirResourceType)]`, and a live safeguard registry (`IHipaaSafeguardCheck`) surfaced at `GET /admin/hipaa/safeguards` via `MapHipaaSafeguardsEndpoint()`. Wired in PDMS, EHR, HIS, HIE, SmartConnect.
- **ClinicianNotification** (`Dialysis.BuildingBlocks.ClinicianNotification` + `.Twilio` SMS / `.Apns` iOS / `.Fcm` Android) — multi-channel clinician alert dispatcher. `AddClinicianNotification()` registers `IClinicianNotificationDispatcher`, which fans a channel-agnostic `ClinicianNotificationRequest` out across registered `IClinicianNotificationSender`s and returns per-channel `ChannelOutcome`s. Used by PDMS on-call escalation (and EHR) to page clinicians with minimal PHI. Senders are registered per host/environment.
- **DurableCommandBus** — opt-in write durability (see dedicated section below).
- **Documents** (blob-store abstractions), **Direct** (Direct secure messaging), **PlatformGateway**, **DistributedCache** (single `Dialysis.BuildingBlocks.DistributedCache.Valkey` project), **Tefca** (German gematik TI building block).

`DomainDrivenDesign/` holds DDD primitives + persistence base classes; `CQRS/` holds CQRS contracts and pipeline behaviors. `Shared/Dialysis.Module.{Contracts,Hosting,Hosting.Testing}` provide host scaffolding consumed by every API `Program.cs` via `builder.AddModuleHost<XPermissionCatalog>(new ModuleHostingOptions { ModuleSlug = "x", HandlerAssemblies = [...] })`; `Shared/Dialysis.Module.Gateway` is the single YARP edge gateway (see "Browser entry point" above).

### Per-context BFFs

Every user-facing context has its own BFF host (`Dialysis.HIS.Bff`, `Dialysis.EHR.Bff`, `Dialysis.PDMS.Bff`, `Dialysis.SmartConnect.Bff`, `Dialysis.HIE.Bff`, `Dialysis.Admin.Bff` [for `identity-web`], `Dialysis.PatientPortal.Bff` [for the portal], plus the OIDC-handshake `Dialysis.Identity.Bff`). All build on `Shared/Dialysis.Module.Bff`:

- `AddModuleBff()` / `MapModuleBff()` — OIDC against Keycloak + path-scoped cookie (e.g. cookie path `/his` so contexts don't collide on the shared `:9090` origin) + the session ticket stored in Valkey + a bearer-forward YARP proxy that attaches the session's access token to `/{ctx}/api/*` and `/{ctx}/hubs/*` calls into the module API. Session continuity is BFF-side via `ITokenRefreshService` (see Identity / auth).
- `Shared/Dialysis.Module.Bff.Events` layers **event-driven push** on top — `AddModuleBffEvents(t => t.AddConsumer<TEvent,TConsumer>())` + `MapModuleBffEvents()` gives a BFF a consume-only Transponder/RabbitMQ subscription (queue `bff-<slug>`, no DbContext/outbox) and a `[Authorize]` SignalR `NotificationsHub` at `/{ctx}/events`. Consumers map an integration event to a PHI-light `BffNotification` and push it via `IBffNotifier` to the `patient:{id}`/`user:{sub}` group; the SPA toasts it and refetches through the synchronous API. Reads stay synchronous — this is real-time push only, not a BFF read model. Wired in EHR/PDMS/HIS BFFs and the PatientPortal BFF today.

### Compliance surfaces (GDPR / BDSG)

Two distinct mechanisms; do not conflate them.

- **Storage limitation (Art. 5(1)(e)) — scheduled purge.** The HIE Documents slice owns the only retention pipeline today: `DocumentRetentionPolicy` per `DocumentReference.Kind`, set via the admin UI at `/hie/admin/documents/retention`. The `RetentionPurgerHostedService` ticks every 24 h and is opt-in via `Documents:Retention:AutoPurge` (default `false`); no windows are seeded, so the purger is a no-op until the DPO adopts policies. Purged documents transition to `EnteredInError` with `StorageRef = purged://…` (tombstone, so audit replay sees deliberate purge, not data loss) and the blob is deleted via `IDocumentBlobStore.DeleteAsync`.
- **Right to erasure (Art. 17) — approve-and-execute pipeline.** Patient files a request → DPO reviews → DPO approves via `/admin/data-protection/data-subject-rights` → `DefaultDataSubjectRightsService.ApproveErasureAsync` walks every registered `IPatientEraser` and persists the per-module breakdown to `IErasureRequestStore`. To participate, a module implements `IPatientEraser` (the contract mirrors `IModuleDataExtractor` for Art. 15 export) and registers it in its composition extension. Today four modules are wired: `HieDocumentsPatientEraser` (HIE Documents — tombstone + blob purge), `HisPatientEraser` (HIS — soft-delete the three Audit-tracked aggregates, hard-delete RA-capability + device-reading rows), `EhrPatientEraser` (EHR — soft-delete the Patient root + chart/scheduling/portal/notes/billing aggregates), and `PdmsPatientEraser` (PDMS — soft-delete sessions and their per-session children by `SessionId`, plus direct patient-linked rows). SmartConnect deliberately doesn't ship one: it routes messages mentioning patients but doesn't own patient PII. Erasers use EF Core 7+ `ExecuteUpdateAsync`/`ExecuteDeleteAsync` so a long-tenured patient with thousands of telemetry rows clears in one round-trip per type.

### Durable command bus (opt-in write durability)

`Dialysis.BuildingBlocks.DurableCommandBus` (+ `.AspNetCore`) moves the durability boundary from "the row is in Postgres" to "the command is in a durable RMQ queue with publisher confirms acknowledged." An opted-in endpoint calls `IDurableCommandBus.EnqueueAsync` instead of `ICqrsGateway.SendCommandAsync`, returns `202 Accepted` with a status-endpoint URL, and `DurableCommandConsumer<TDbContext>` (an `IConsumer<DurableCommandEnvelope>`) dequeues, opens an explicit EF transaction, claims via the `command_ledger` row (idempotent on `CommandId`), dispatches into the existing `ICommandHandler` via the registered closure on `IDurableCommandCatalog`, marks the ledger row Applied, and commits. Handler change + ledger row commit in one transaction; if anything throws, the tx rolls back and the broker redelivers. CLAUDE.md's "no event sourcing" rule holds — the queue is an in-flight buffer, never a permanent log; aggregates still persist current state via EF.

Today two slices opt in: **PDMS `RecordReading`** (flag `Pdms:DurableCommands:RecordReading:Enabled`, default off) and **HIS `IngestDeviceReading`** (flag `His:DurableCommands:IngestDeviceReading:Enabled`, default off) — both telemetry-shape writes where the durability ask is sharpest. Both use the id-from-CommandId trick (`reading.Id = command.CommandId` / `DeviceReadingRecord.Id = command.ReadingId`) so 202 callers know the new row's id without polling. Status surface: `GET /api/v1.0/command-status/{correlationId}` returns the ledger row; authorized against the row's stored `requestedBySubject` so a leaked correlation id can't probe across permission boundaries. SPA side: a `useDurableCommand` hook wraps any opt-in write — enqueue → poll status endpoint → fire toasts → expose `tracking` state for inline progress chips.

Observability: the bus emits a `Dialysis.DurableCommandBus` meter (`commands_enqueued` / `commands_applied` / `commands_failed` counters; `latency` + `enqueue_latency` histograms) — added to `ModuleTelemetryOptions.AdditionalMeters` in each module that calls `AddDurableCommandBus<>`. Grafana dashboard JSON at `deploy/k8s/observability/dashboards/durable-command-bus.json`; alert rules at `deploy/k8s/observability/alerts/durable-command-bus.yaml`.

PDMS uses **TimescaleDB** for the highest-volume write path: `pdms_sessions.IntradialyticReadings` is a hypertable partitioned by `ObservedAtUtc` with a 7-day compression policy + 365-day retention. Same EF Core driver, same wire protocol, transparent to application code; only the Postgres image differs (`timescale/timescaledb:latest-pg17` on PDMS, stock `postgres:17-alpine` elsewhere). Aspire AppHost + the `pg-pdms` CNPG cluster CR + the EF migration handle the wiring.

**Both tiers of the durability story go HA via Kubernetes operators** (`deploy/k8s/operators/`). CloudNativePG `Cluster` CRs render per module Postgres with sync replication on the clinical tier (HIS/EHR/PDMS) and async on the integration tier (SmartConnect/HIE), plus WAL archiving to S3/MinIO for PITR. A RabbitMQ Cluster Operator `RabbitmqCluster` runs the broker as a 3-replica quorum-queue cluster; `TransponderRabbitMqOptions.QueueType = Quorum` declares the durable-command queue Raft-replicated. PgBouncer sidecars front each module's Postgres in transaction-pooling mode. Per-env knobs live in `deploy/k8s/operators/values/{dev,staging,prod}.env`; apply with `./deploy/k8s/operators/render.sh <env> | kubectl apply -n dialysis-<env> -f -`. Aspire's k8s publisher emits the application-layer chart; the operator CRs are hand-maintained because Aspire's `KubernetesEnvironment` doesn't model arbitrary CRDs.

### Module boundary invariant (enforced)

`tests/Dialysis.ArchitectureTests` codifies, via assembly-reference inspection + `NetArchTest`:

> A project under module `X` may only reference its own siblings, the shared layers (`Dialysis.DomainDrivenDesign`, `Dialysis.BuildingBlocks`, `Dialysis.CQRS`, `Dialysis.Module.Contracts`, `Dialysis.Module.Hosting`), and the **`Dialysis.Y.Contracts`** assembly of any other module. Anything else is a build-test failure.

It also enforces aggregate-root encapsulation and integration-event versioning. Cross-module functionality goes through an integration event in `<Module>.Contracts` + an `IConsumer<>` on the receiving side (or, for synchronous reads, a cross-module query like HIE's consent check) — never a direct project reference.

### Identity / auth

JWT Bearer is registered only when `<Module>:Authentication:Authority` is set; in Development with no Authority, `ICurrentUser` exposes all permissions for local work. IdP role/group names map to module permission strings via `<Module>:Authentication:RolePermissionMap`. Patient-portal-facing endpoints additionally filter by patient claim (e.g. `his_patient_id` or `sub` matching route `patientId`). The Identity BFF + Keycloak realm `dialysis` are the canonical IdP — see `src/backend/Identity/RUNBOOK.md` and `ARCHITECTURE.md`.

**Multi-IdP federation (Okta / Auth0 / Entra) goes through Keycloak brokering, not BFF-side IdP swapping.** Keycloak stays the only direct OIDC client; upstream IdPs are added as realm `identityProviders[]` entries (placeholders for `okta`/`auth0`/`entra` ship disabled in `dialysis-realm.json`) and surfaced to the SPA through `IIdentityProviderCatalog` + `GET /identity/providers`. The BFF forwards `kc_idp_hint=<alias>` on the OIDC auth request when the caller hits `/identity/login?provider=<alias>`, gated by an allowlist so unknown aliases can't probe Keycloak. See RUNBOOK §8.

**Session continuity is handled BFF-side, not by re-login.** `ITokenRefreshService` is wired to the cookie handler's `OnValidatePrincipal` event: when the saved `expires_at` token falls within 60 s of now, the BFF calls Keycloak's `/token` endpoint with the stashed `refresh_token`, rewrites `access_token`/`refresh_token`/`id_token`/`expires_at` on the auth ticket, and sets `ShouldRenew = true`. Failures (no refresh token, Keycloak rejects the grant) call `RejectPrincipal()` so the SPA bounces back through login instead of sitting on a stale session. The OIDC handler requests `offline_access` scope on auth to make sure Keycloak issues a refresh token in the first place.

**Permission claims reach the SPA via `dialysis_permission`.** The Keycloak realm mapper emits a JSON-typed `dialysis_permission` claim through the userinfo endpoint; the BFF's `/identity/user` response parses it into a top-level `permissions: string[]` array. The SPA's `PermissionGate` does a simple `permissions.includes(required)` check — call sites pass typed permission strings from the module's `<Module>Permissions` catalog, no claim plumbing required.

### Frontend (per-module SPAs)

The SPA tier mirrors the backend boundaries: **one independent Vite/React app per user-facing context** under `src/frontend/` — `his-web`, `ehr-web`, `pdms-web`, `smartconnect-web`, `hie-web`, `identity-web` (admin console, served at `/admin`), `patient-portal-web` (served at `/portal`). Each app is standalone (React 18, Vite 6, TypeScript, TanStack Query, React Router, Axios, SignalR client, Tailwind; charts via ECharts/D3; PDFs via pdfjs; flow diagrams via XYFlow). There is **no** npm workspace and **no** shared UI package — instead a handful of cross-cutting files are duplicated byte-for-byte across the apps that need them: `src/shared/lazyPage.ts` (a `React.lazy` wrapper for named exports), the auth shell, the `useDurableCommand` client, the toast/theme shells, `eslint.config.js`, and `tsconfig.json` in all seven, plus the `src/features/patients/` helpers across the ehr/pdms/identity subset (see the Build/run/test §Frontend note for the full inventory and the intentional hie/portal divergences). Editing one copy means updating the others by hand; `tools/frontend/check-duplicate-sync.sh` (wired as the `duplicate-sync` job in `frontend.yml`) fails CI on drift.

- Each app talks **only** to its own context BFF through the Gateway: API at `/{ctx}/api`, SignalR hubs at `/{ctx}/hubs` (synchronous data) and `/{ctx}/events` (event-driven push), auth at `/{ctx}/identity`. Vite dev proxies these (plus `/auth`) to `VITE_GATEWAY_URL` (default `http://localhost:9090`).
- Pages are loaded through `lazyPage` so each route ships in its own chunk and only downloads on first visit; the router wraps the outlet in `<Suspense>`.
- `humanizeError` maps ProblemDetails / network errors to user-readable sentences — never expose raw status codes or stack traces to clinical users.
- **TanStack Query mutation pattern** — queue/chart/note mutations use optimistic-update + rollback on error + invalidate-on-settle. Copy the canonical `useQueueMutation` shape for new mutations.
- Lint/format is strict: `eslint . --max-warnings=0` + Prettier, enforced by a Husky `pre-commit` + `lint-staged`; the shared flat `eslint.config.js` wires TS + React + React Hooks + React Refresh + `eslint-plugin-jsx-a11y` (recommended, accessibility). CI runs the per-app eslint/prettier/tsc/vitest/build matrix in `frontend.yml`.

## Conventions & tooling

- **Warnings are errors.** `Directory.Build.props` sets `TreatWarningsAsErrors=true`, `EnforceCodeStyleInBuild=true`, `GenerateDocumentationFile=true` (XML-doc warnings CS1591/1573/1574/1734/1735 are suppressed; missing docs on genuinely public API still surface as other diagnostics). Microsoft VS Threading analyzers are referenced solution-wide; **`VSTHRD200` (async methods must end with `Async`) is pinned to error** in `.editorconfig` — keep new async methods suffixed.
- **Style** (`.editorconfig`, mostly warning-severity so they fail the build): file-scoped namespaces; no `this.`/`Me.` qualification; predefined type keywords; accessibility modifiers required; `readonly` where possible. SonarAnalyzer.CSharp runs in-build (referenced solution-wide in `Directory.Build.props`, so the `.editorconfig` Sonar profile is enforced on every compile — see `docs/reports/sonarqube-dotnet-2026-06-10.md` for the profile rationale). Noisy rules are tuned down (`CA2007`, `CA1303` off; `CA1062`, `CA1848` suggestion).
- **SonarQube MCP** (`.mcp.json` + `.github/instructions/sonarqube_mcp.instructions.md`): when modifying code, disable automatic analysis at task start and re-enable it (and run `analyze_file_list` on changed files) at the very end if those tools exist. Look up project keys via `search_my_sonarqube_projects` (don't guess); don't re-query issues immediately after a fix (the server lags).
- **CI**: six workflows, build/test driven by the NUKE build. `build-test.yml` runs `./build.sh Test Pack` over `Dialysis.slnx` (full build + every module/buildingblocks test + the architecture gates in one job; a `workflow_dispatch`-only `his-outbox-e2e` job covers the SQL+RabbitMQ golden path); `deploy-artifacts.yml` regenerates the compose/Helm artifacts from the AppHost and fails on drift (`--deploy false` + `git diff --exit-code`); `frontend.yml` (per-app eslint/prettier/tsc/vitest/build matrix); `security.yml` (dotnet/npm dependency audit, Trivy, OWASP ZAP); `docs-regulatory.yml` (SmartConnect PDF source-of-truth on push/PR + the daily HIPAA regulatory-feed drift check); `load-test.yml` (scheduled/dispatch k6 scenarios). Static analysis is owned by the Aspire-hosted SonarQube server (`tools/sonarqube/`) — CodeQL was removed once SonarQube took over the security + quality role. `tools/sonarqube/` also ships CLI helpers: `bootstrap.sh` (first-boot provisioning, invoked by the AppHost's `sonarqube-bootstrap` container), `scan.sh` (`sonarscanner begin` → build + coverage → `sonarscanner end`, pushing results to the dashboard), and `fix.sh` (locally auto-applies mechanically-fixable analyzer issues via `dotnet format`, clears stale `.sonarqube/`, and rebuilds Release to verify).
